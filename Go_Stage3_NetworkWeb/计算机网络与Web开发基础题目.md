### 📝 计网测试（附答案）

**Q1：【场景题 - TCP 挥手】如果你的聊天服务器突然断电或者网线被拔了，底层的 TCP 四次挥手还能挥得起来吗？客户端会立刻知道服务端掉线了吗？**

- **答案：** 绝对挥不起来！四次挥手是需要双方在网络通畅的情况下互相发送 `FIN` 和 `ACK` 报文的。如果直接拔网线或断电，操作系统根本来不及发送任何断开信号。
- **结果：** 客户端**不会立刻知道**。客户端的 TCP 连接在它自己的系统里看起来依然是“建立状态”。直到客户端尝试给服务端发送消息，等了很久发现收不到 `ACK` 确认（超时），或者底层的 `Keep-Alive` 保活机制探测失败，它才会恍然大悟：“哦，原来服务器早就挂了”。

**Q2：【选择题 - HTTP 请求】如果前端要传一段包含 2000 个字、甚至带点表情包的超长聊天记录给你的 Go 服务器，你设计的 API 应该用 GET 还是 POST？为什么？**

- **答案：** 必须用 **POST**。
- **原因：** GET 请求的参数是拼接在 URL 里的（`?msg=xxx`）。虽然 HTTP 协议本身没限制 URL 长度，但**浏览器和 Web 服务器（如 Nginx）**通常会限制 URL 长度（一般在 2KB 到 8KB 左右）。2000 个字加上 URL 编码绝对会超长被截断报错。而 POST 请求的数据放在 **请求体 (Body)** 中，没有严格的长度限制，且不会暴露在浏览器的地址栏历史记录里，更适合提交长文本。

**Q3：【定位题 - 状态码】如果你写的 Go 接口代码里出现了数组越界导致 `panic`（程序崩溃），前端调用你这个接口时，大概率会收到哪个 5XX 的状态码？如果是你主动拦截了没权限的用户呢？**

- **答案：**
  - 代码崩溃（`panic`）：通常会返回 **`500 Internal Server Error`**（服务器内部错误）。表示服务器代码写出 bug 了。
  - 主动拦截无权限用户：应该返回 **`401 Unauthorized`**（未登录验证）或 **`403 Forbidden`**（已登录但权限不足）。4XX 代表是客户端的问题（没带对令牌）。

**Q4：【拔高题 - HTTPS 安全】既然对称加密（AES）速度极快，非对称加密（RSA）速度很慢，为什么 HTTPS 不直接全程只用对称加密来传输数据？**

- **答案：** 因为**“密钥分发难题”**。如果只用对称加密，服务端怎么把这把“解密钥匙”安全地传给浏览器呢？如果在网络上直接明文发钥匙，黑客截获了钥匙，后续的加密就形同虚设了。
- **HTTPS 的精妙配合：** 所以必须先用非对称加密，把这把“对称钥匙”安全地送到浏览器手里（因为非对称加密的公钥不怕被截获）。一旦双方都拿到了只有彼此知道的“对称钥匙”，后续的海量聊天数据就可以用对称加密高速传输了。
