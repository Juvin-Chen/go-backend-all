# Go 语言 panic 

> 定位：Go 内置机制，用于处理致命、不可恢复的问题。
> 适用场景：程序启动失败、代码逻辑崩溃（空指针/越界）、配合 recover 做服务保护。

## 一、基础概念

### 什么是 panic

panic 是 Go 的内置函数，用于主动触发程序异常流程。
它不是普通 error，而是“当前流程无法继续”的信号。

### 核心特性

| 特性 | 说明 |
| :--- | :--- |
| 处理问题 | 严重、无法继续运行的问题（不等于普通业务错误） |
| 触发效果 | 当前函数后续代码停止执行 |
| 恢复机制 | 可配合 recover 捕获，避免进程直接退出 |
| 作用范围 | 默认只对当前 goroutine 调用栈生效 |

### 基本写法

```go
panic("错误信息")
recover()
```

## 二、panic 执行过程

### 触发方式

```go
panic("程序发生致命错误，无法继续")
panic(123)
panic(errors.New("错误对象"))
```

### 执行效果

触发 panic 后会发生：

1. 立即停止当前函数后续代码。
2. 逆序执行当前函数 defer（LIFO）。
3. 将 panic 向上层调用者继续抛出。
4. 若到达最顶层仍未恢复，程序崩溃退出。
5. 控制台打印 panic 信息与调用栈。

## 三、panic 与 error 区别

| 特性 | error（普通错误） | panic（致命崩溃） |
| :--- | :--- | :--- |
| 处理问题 | 可预料、可恢复的业务问题 | 不可预料、无法继续的致命问题 |
| 程序是否崩溃 | 否 | 默认会 |
| 处理方式 | 判断 err != nil | 用 recover 捕获 |
| 使用场景 | 参数错误、查询失败、表单校验 | 配置损坏、空指针、越界、启动失败 |

一句话：error 是可处理问题，panic 是不可继续问题。

## 四、recover：panic 的恢复机制

### 关键规则

1. recover 只能在 defer 函数中生效。
2. 不在 defer 中调用 recover，返回 nil，无法捕获 panic。
3. 捕获后可把崩溃转换为可控错误响应（例如 Web 返回 500）。

### 正确与错误示例

```go
// 正确：defer 中调用 recover
func test1() {
    defer func() {
        if err := recover(); err != nil {
            log.Println("捕获崩溃:", err)
        }
    }()

    panic("空指针异常")
}

// 错误：直接调用 recover（无效）
func test2() {
    _ = recover()
    panic("空指针异常")
}
```

## 五、没有 recover 会怎样

### 示例

```go
func main() {
    panic("程序崩溃了")
    fmt.Println("这行不会执行")
}
```

程序会退出，并打印 panic 信息和调用栈。

### Web 服务中的影响

- 无 Recovery 中间件：某个请求 panic 可能导致整个服务退出。
- 有 Recovery 中间件：该请求返回 500，服务进程保持可用。

## 六、Recovery 中间件捕获范围

### 结论

Recovery 可以捕获：

- 最终 handler 的 panic。
- Recovery 内层中间件的 panic。
- handler 调用链中的 panic。

Recovery 不能捕获：

- 其他 goroutine 的 panic。
- os.Exit/log.Fatal 导致的退出（不会执行 defer）。

### 注册顺序建议

Recovery 要放在中间件链最外层。

```go
handler := middleware.Recovery(
    middleware.Logger(
        middleware.RequestID(mux),
    ),
)
```

## 七、log.Fatal 与 panic 的关系

常见误区：log.Fatal 底层不是 panic，而是 os.Exit(1)。

```go
// log.Fatal 等价行为：打印后直接退出，不执行 defer
func Fatal(v ...any) {
    std.Output(2, fmt.Sprint(v...))
    os.Exit(1)
}
```

| 特性 | panic | log.Fatal |
| :--- | :--- | :--- |
| defer 是否执行 | 会 | 不会 |
| recover 是否可捕获 | 可以 | 不可以 |
| 退出码 | 常见为 2 | 1 |

建议：

- 启动阶段不可恢复错误：可用 log.Fatal。
- 运行阶段请求处理：优先 error + Recovery。

## 八、实战建议

### 推荐使用 panic 的场景

- 启动阶段不可恢复错误（配置/模板/数据库初始化失败）。
- 程序内部不变量被破坏（理论上不应到达的分支）。
- 开发调试阶段快速暴露严重问题。

### 不建议使用 panic 的场景

- 用户输入校验。
- 业务规则错误（余额不足、权限不足）。
- 外部依赖暂时失败（网络超时、API 错误）。

这些都应该返回 error 并做正常错误处理。

## 九、可直接使用的中间件示例

```go
package middleware

import (
    "context"
    "fmt"
    "log"
    "net/http"
    "time"
)

type contextKey string

const RequestIDKey contextKey = "requestID"

func Logger(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        log.Printf("Started %s %s", r.Method, r.URL.Path)
        next.ServeHTTP(w, r)
        log.Printf("Completed %s in %v", r.URL.Path, time.Since(start))
    })
}

func Recovery(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {
            if err := recover(); err != nil {
                log.Printf("Panic recovered: %v", err)
                http.Error(w, "Internal Server Error", http.StatusInternalServerError)
            }
        }()

        next.ServeHTTP(w, r)
    })
}

func RequestID(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        id := fmt.Sprintf("%d", time.Now().UnixNano())
        ctx := context.WithValue(r.Context(), RequestIDKey, id)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}
```

## 十、常见坑位清单

| 坑点 | 正确做法 |
| :--- | :--- |
| 业务错误直接 panic | 业务错误用 error 返回 |
| recover 不放 defer | recover 必须写在 defer 函数里 |
| Recovery 放内层 | Recovery 放最外层 |
| goroutine 内不处理 panic | 每个 goroutine 自己做 defer recover |
| 把 log.Fatal 当可恢复错误处理 | log.Fatal 会直接退出进程 |

## 十一、终极总结

1. panic 用于“程序无法继续”的致命问题。
2. error 用于“业务可处理”的普通问题。
3. recover 只能在 defer 中使用。
4. Web 服务必须有 Recovery 中间件，且放外层。
5. goroutine 的 panic 需要在各自 goroutine 内单独 recover。
6. log.Fatal 不是 panic，不能被 recover 捕获。
