# 编程中「视图（View）」
适用场景：Go 结构体返回、数据库、前端、操作系统等全领域

## 一、一句话终极定义
**视图 = 原始数据的访问窗口 / 展示形式**
不复制数据、不存储数据，只提供一种“查看/使用”数据的方式。

## 二、视图三大核心特征
1. **不拷贝数据**：始终指向原始数据，不占用额外内存。
2. **不存储数据**：本身不持久化，依赖原始数据源。
3. **可控权限**：常设计为“只读”，防止意外修改原始数据。

## 三、不同场景下的「视图」含义
### 1. Go 后端：数据只读视图
- 场景：结构体数据不想被外部修改，又不想完整拷贝。
- 含义：返回一个**只能读、不能改**的包装结构/接口。
- 对比：
  - 副本：复制一份新数据，安全但占内存、有性能开销。
  - 视图：直接看原数据，高效安全。

### 2. 数据库视图
- 是一张**虚拟表**，不存真实数据。
- 只是一条查询语句的封装，查询时实时从真实表读取。
- 作用：简化查询、隐藏敏感字段。

### 3. 前端 / MVC 中的 View
- 对应 HTML 页面。
- 只负责展示后端数据，不存储、不复制数据。

### 4. 操作系统视图
如文件视图、内存视图。
- 只是对底层数据的一种展示方式。
- 操作视图等价于操作原始数据，但界面更友好。

## 四、副本 vs 视图（关键区别）
| 方式 | 数据拷贝 | 内存占用 | 性能 | 安全性 |
|------|----------|----------|------|--------|
| 数据副本 | 复制一份新数据 | 高 | 低 | 安全，互不影响 |
| 数据视图 | 不拷贝，指向原数据 | 低 | 高 | 只读更安全 |

## 五、总结
1. 视图 = 窗户，原始数据 = 窗外风景。
2. 视图只负责“看/访问”，不创造、不复制数据。
3. 数据量大时，用**只读视图**替代副本，兼顾性能与安全。
