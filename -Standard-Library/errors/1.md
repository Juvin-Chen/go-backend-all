# Go 语言错误处理

适配场景：留言板项目、标准库使用、Gin 框架前置基础

核心解决：**什么是包 / 错误 / 错误调用** + **全场景错误处理方案** + **所有标准库通用套路**

------

## 一、核心基础概念

### 1. 什么是包（Package）？

Go 语言的**代码模块化单位**，相当于「工具箱」，所有功能都封装在包里。

- 官方自带：**标准库**（`net/http`/`html/template`/`log`/`errors`/`os`）
- 自己写的：`store`/`handlers`/`middleware`
- 作用：不同包负责不同功能，报错来源 = 对应工具箱出问题

### 2. 什么是错误（error）？

程序执行**失败 / 异常**时返回的「提示信息」，是 Go 统一的标准类型。

- 所有标准库、自定义函数的错误，**本质完全一样**
- 自带 `Error()` 方法，自动输出文字描述

### 3. 什么是错误调用 / 处理？

1. 调用函数 → 函数失败返回 `err`

2. 判断 

   ```
   err != nil
   ```

    → 执行打印 / 返回 / 终止等操作

   这就是错误处理，是后端开发的必备操作。

------

## 二、Go 错误底层：统一的 `error` 接口

Go 语言规定：**全语言所有错误，都实现这个接口**，没有例外！

```
// 所有错误的"祖宗"，统一标准
type error interface {
    Error() string // 唯一作用：返回错误文字
}
```

### 配套工具：`errors` 标准库

仅一个作用：**快速创建自定义错误**，不是存储所有错误的仓库！

```
import "errors"

// 创建一个错误
err := errors.New("昵称不能为空，提交失败")
```

## 三、全场景错误处理方案

我按照项目中**最常遇到的场景**整理，所有标准库通用。

### 场景 1：通用业务逻辑错误（最基础）

适用：自定义判断、参数校验、业务规则

```
// 1. 调用函数/判断逻辑，接收错误
nickname := ""
if nickname == "" {
    // 2. 创建错误
    err := errors.New("昵称不能为空")
    // 3. 处理错误：打印日志
    log.Println("业务错误：", err)
    return
}
```

------

### 场景 2：Web/HTTP 请求错误（web项目核心）

适用：路由、请求方法、表单解析、接口处理

**固定套路 + 固定函数 `http.Error()`**

```
// 1. 判断请求方法
if r.Method != http.MethodPost {
    // 专用错误：返回给浏览器 + 设置状态码
    http.Error(w, "请求方式不允许", http.StatusMethodNotAllowed)
    return
}

// 2. 解析表单错误
err := r.ParseForm()
if err != nil {
    log.Println("表单解析失败：", err)
    http.Error(w, "请求参数错误", http.StatusBadRequest)
    return
}
```

#### ✅ `http.Error()` 固定参数（记）

```
http.Error(
    w http.ResponseWriter,  // 固定：给浏览器响应
    err string,             // 必须是字符串：err.Error()
    code int                // HTTP状态码：400/404/500
)
```

------

### 场景 3：模板渲染错误（html/template）

适用：HTML 页面加载、渲染数据

```
// 加载模板
tmpl, err := template.ParseFiles("templates/index.html")
if err != nil {
    // 1. 打印服务器日志（黑匣子）
    log.Printf("模板加载失败：%v", err)
    // 2. 返回浏览器500错误
    http.Error(w, "服务器页面加载失败", http.StatusInternalServerError)
    return
}

// 渲染数据
err = tmpl.Execute(w, data)
if err != nil {
    log.Println("页面渲染失败：", err)
    http.Error(w, "页面渲染失败", 500)
    return
}
```

### 场景 4：文件 / 静态资源错误（os 包）

适用：静态文件、配置文件、日志文件

```
file, err := os.Open("static/style.css")
if err != nil {
    log.Println("静态文件加载失败：", err)
    http.Error(w, "静态资源不存在", 404)
    return
}
```

### 场景 5：并发安全存储错误（store 包）

适用：你的 `MemoryStore` 增删改查

```
func (s *MemoryStore) Delete(index int) error {
    s.mu.Lock()
    defer s.mu.Unlock()

    // 业务错误：下标越界
    if index < 0 || index >= len(s.messages) {
        return errors.New("留言不存在，删除失败")
    }

    s.messages = append(s.messages[:index], s.messages[index+1:]...)
    return nil
}

// 调用时处理
err := store.Delete(10)
if err != nil {
    log.Println("删除失败：", err)
    http.Error(w, err.Error(), 400)
    return
}
```

## 四、所有标准库统一错误套路（背会这 4 行）

**不管哪个包（http/template/os/log），处理方式完全一样！**

```
// 1. 执行操作，接收返回值和错误
result, err := 包.函数()

// 2. 判断是否出错
if err != nil {
    // 3. 处理错误：打印日志 + 返回前端
    log.Println("错误信息：", err)       // 服务器记录（必写）
    http.Error(w, err.Error(), 状态码)   // Web项目专用
    return                              // 终止函数（必写）
}

// 4. 无错误，执行业务逻辑
```

------

## 五、错误打印工具对比（项目中该用哪个？）

| 打印方式           | 作用               | 适用场景             | 项目推荐度 |
| :----------------- | :----------------- | :------------------- | :--------- |
| `fmt.Println(err)` | 普通打印           | 临时调试、练习题     | ❌ 不推荐   |
| `log.Println(err)` | 带时间戳、并发安全 | 服务器日志（黑匣子） | ✅ 必须用   |
| `http.Error()`     | 把错误返回给浏览器 | Web 接口、页面请求   | ✅ 专用     |

------

## 六、HTTP 状态码对应错误（直接用）

你只需要记这 4 个，覆盖所有场景：

1. `200`：请求成功 
2. `400`：用户参数错误（前端问题）
3. `404`：页面 / 文件不存在
4. `500`：服务器代码错误

------

## 七、新手必避坑（关键！）

1. **所有包的错误类型统一**：不存在「http 专属错误」「模板专属错误」
2. **`http.Error(x,x,x)参数固定` 只能传字符串**：不能传结构体，必须用 `err.Error()`
3. **出错必须 `return`**：不终止函数会继续执行，导致程序崩溃
4. **`log` 是项目必备**：`fmt` 仅用于临时调试，不能上项目
5. **`errors` 包只用来创建错误**：不负责处理 / 打印

------

## 八、总结（一句话记住）

1. **包 = 工具箱**，报错 = 工具箱出问题
2. **错误 = 统一的 `error` 类型**，所有标准库用法一样
3. **处理套路 = 判断 err → 打印日志 → 返回前端 → 终止函数**
4. **项目只用 `log` 打印，Web 用 `http.Error` 返回浏览器**
